# ADR-0009: 지원 플랫폼 범위 — Linux는 후속으로 defer

- 날짜: 2026-07-23 / 상태: 승인

## 맥락 (무엇을 결정해야 했나)

PRD는 여러 곳(§0, §1.1, §5.2 "MVP에 반드시 포함할 기능 > 1. 데스크톱 앱 실행", §8.2)에서 macOS·Windows·Linux 세 플랫폼 실행을 명시한다. RW-11(지원 플랫폼 릴리스 검증, `docs/planning/remaining-work.md`)을 착수하며 실제로 검증 가능한 범위를 정해야 했다.

제약:
- 1인 개발 체제이고, 개발·검증 가능한 실기기는 macOS(arm64, 이 세션의 환경) 뿐이다. Windows는 GitHub Actions 러너로 빌드까지는 자동화했으나(#32) 트레이·단축키·Credential Manager 같은 실제 GUI 동작은 여전히 사람이 직접 확인해야 한다. Linux는 빌드조차 아직 CI에 없다(GitHub Actions에 job을 추가하는 것 자체는 어렵지 않으나, Tauri의 Linux 빌드는 `webkit2gtk` 등 시스템 패키지 의존성이 있어 검증된 적이 없다).
- Linux 배포판을 만들어도 실제 설치·트레이·단축키 동작을 확인할 기기가 없어, "빌드된다"와 "동작한다"의 간극이 macOS/Windows보다 크다.
- RW-11의 완료 조건은 "지원한다고 문서화한 모든 OS/arch에 설치 가능한 산출물과 smoke 기록이 있을 것"이므로, 검증 수단이 없는 플랫폼을 지원 범위에 남겨두면 이 조건을 영구히 못 채운다.

## 결정

**Linux 지원을 v0.1.x 릴리스 범위에서 제외하고, PRD의 관련 서술을 "후속(미정)"으로 명시적으로 축소한다.** GitHub Actions에 Linux 빌드 job을 추가하는 것도 지금은 하지 않는다 — job이 있어도 실행 검증 수단이 없어 "빌드는 되는데 동작은 모르는" 상태만 만들고, 이는 검증되지 않은 산출물을 릴리스에 올리는 것과 실질적으로 같다.

지원 범위는 **macOS(arm64/x86_64) + Windows 11**로 한정한다. RW-11은 이 두 플랫폼만 검증 대상으로 진행한다.

## 근거

- 코드 자체는 Linux를 막지 않는다(Tauri 2·Go 둘 다 Linux 크로스컴파일/네이티브 빌드 가능) — 이번 결정은 "당장 검증 못 하는 것을 지원한다고 문서에 써두지 않는다"는 문서 정직성 문제이지, 아키텍처상 Linux를 배제하는 결정이 아니다. 필요해지면(Linux 사용자 요청, 검증 가능한 기기 확보 등) 재개한다.
- 검증 안 된 플랫폼을 "지원"이라 적어두면 사용자가 설치했다가 실패했을 때 신뢰가 깎이고, RW-11/backlog #29 같은 완료 조건이 영원히 열린 채로 남는다.

## 결과·트레이드오프

- PRD §5.2 "1. 데스크톱 앱 실행"에 이 ADR을 참조하는 각주를 추가하고, §0/§1.1/§8.2의 장기 비전 서술(Linux 포함)은 그대로 둔다 — 이 결정은 **v0.1.x 릴리스 범위**의 축소이지, 제품의 장기 목표를 철회하는 것이 아니다.
- `docs/planning/remaining-work.md` RW-11, `docs/planning/backlog.md`, README에도 "Linux는 이 ADR로 defer"를 반영한다.
- 재개 조건: Linux 사용자 요청이 실제로 발생하거나, 검증 가능한 Linux 기기/CI 러너 기반 실제 설치 검증 경로가 확보되면 이 ADR을 대체하는 새 ADR로 재개를 결정한다.
