# ADR-0002: Tauri 2 + Go sidecar 아키텍처

- 날짜: 2026-07-03 (2026-07-04 근거 보강) / 상태: 승인

## 맥락
macOS/Windows/Linux 3개 OS에서 동작하는 **트레이 상주 데스크톱 앱**이 필요하다 (PRD §1.1, §5.2-1). 핵심 UX 제약(PRD §5, §10):
1. 시스템 트레이에 상주
2. 앱이 포커스되지 않아도 동작하는 **글로벌 단축키**로, 다른 앱에서 드래그·선택한 텍스트를 즉시 검색
3. 백그라운드에서 AI 검색을 돌리고, 결과가 오면 **시스템 알림**으로 통지
4. macOS·Windows 우선 지원 (Linux는 후순위)
5. 라이선스는 무료·비독점(허용 목록: `docs/rules/development-cycle.md`)

즉 "창 안의 UI"보다 **"창 밖의 OS 통합"(트레이·전역 단축키·알림)** 이 프레임워크 선택의 실질적 결정 요인이다. 후보는 아래 "대안 검토" 참조.

## 결정
Tauri 2(UI, 트레이, 글로벌 단축키, 알림 담당) + Go sidecar(`desktopd`, 비즈니스 로직·SQLite·AI 호출·복습 스케줄 담당)로 분리한다. 둘은 로컬 HTTP API로 통신한다 (PRD §7.1, §8.1, §15).

## 근거
- Go: 단일 바이너리 배포, SQLite/HTTP 표준 라이브러리 성숙도, 크로스 컴파일 용이 — 3-OS 배포에 유리
- Tauri: Electron 대비 바이너리 크기·메모리 사용량이 작고, **트레이·글로벌 단축키·알림 OS API 바인딩이 코어/공식 플러그인으로 기본 제공**됨 (요구사항 1·2·3을 프레임워크가 직접 커버)
- Rust 단일 스택(순수 Tauri)도 고려했으나, 비즈니스 로직(복습 스케줄링, AI 파이프라인)을 Go로 작성하는 편이 향후 중앙 서버(`apps/api`, PRD §16)와 코드·팀 지식을 공유하기 쉬움

## 대안 검토 (2026-07-04 보강)
"Go를 메인 언어로" 쓰려면 Go 네이티브 프레임워크인 Wails가 더 직관적이라는 지적이 있어, 요구사항 대비 재검토했다.

| 요구사항 | **Tauri 2 + Go sidecar (채택)** | Wails v2 | Wails v3 | Electron |
|---|---|---|---|---|
| 시스템 트레이 | 코어 내장 | **내장 없음** (3rd-party `systray` 수동 통합) | 내장 | 내장 |
| 글로벌 단축키 | 공식 플러그인 | **내장 없음** (`golang.design/x/hotkey` 등) | 문서상 불명확 | 내장 |
| 시스템 알림 | 공식 플러그인 | Go 라이브러리(beeep 등) | 개선됨 | 내장 |
| 백그라운드 검색 | Go sidecar 전담 | Go 네이티브 | Go 네이티브 | Node |
| 안정성 | **정식 stable** | stable | **알파** (2026-07 기준 정식판 없음) | stable |
| 바이너리 크기 | 작음 | 작음 | 작음 | 큼(Chromium 번들) |
| 라이선스 | MIT/Apache | MIT | MIT | MIT |

- **Wails v2 기각**: 이 앱의 핵심 UX인 트레이·글로벌 단축키가 둘 다 내장이 아니다. 3rd-party 트레이 라이브러리는 Wails 스레드 모델과 충돌해 앱이 멈출 수 있다는 알려진 함정이 있어, 1인 개발에 리스크가 크다.
- **Wails v3 기각(현시점)**: 트레이·알림은 해결됐으나 아직 알파(v3.0.0-alpha.93)라, 프레임워크 자체 버그를 우리가 떠안게 된다. 글로벌 단축키 지원도 불명확. 정식 릴리스 후 재평가 여지는 있음.
- **Electron 기각**: 요구사항은 모두 충족하나 Chromium 번들로 인한 크기·메모리가 트레이 상주 앱(항상 켜둠)에 불리하고, 로직을 Node로 쓰면 Go 메인 방향과 어긋난다.
- **"드래그한 텍스트 즉시 검색"은 프레임워크가 아니라 OS 레벨 문제**다. 실용적 경로는 "글로벌 단축키 → `Cmd/Ctrl+C` 흉내 → 클립보드 읽기"이며(PRD 클립보드 우선 방침), 이 때문에 **글로벌 단축키 내장 여부**가 더 결정적이 되어 Tauri 우위를 강화한다. 선택 영역 직접 읽기(macOS Accessibility / Windows UI Automation)는 권한·복잡도 때문에 MVP 이후로 미룬다.

결론: "Go 메인" 목표는 sidecar 구조로 이미 달성되며(로직 전부 Go, Rust는 창·트레이·단축키 얇은 껍데기), 핵심 UX 요구사항에서 Tauri가 Wails보다 성숙하므로 ADR-0002 결정을 유지한다.

## 결과·트레이드오프
- 장점: UI와 로직의 언어/책임이 명확히 분리, desktopd는 UI 없이도 단독 테스트·실행 가능
- 트레이드오프: 프로세스 2개(UI+sidecar) 생명주기 관리가 추가된다 — UI 종료 시 sidecar도 함께 종료되도록 부모-자식 프로세스 관리가 필요 (백로그 #13에서 구체화)
