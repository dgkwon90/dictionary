# AI 협업·토큰 효율 규칙

Claude(오케스트레이터) + codex(구현 관점) + agy(분석 관점)를 쓰되, 토큰을 아끼는 것이 기본값이다.
1인 개발이라 사람 리뷰어가 없다 — codex/agy 교차 검토가 실질적인 2차 검증 수단이므로, ADR급 결정에는 반드시 돌린다.

## 역할 분담
| 엔진 | 맡기는 것 | 맡기지 않는 것 |
|---|---|---|
| Claude (메인) | 계획·중재·종합, 최종 판단, 코드 작성 주력 | — |
| codex | 구현 타당성 리뷰, 기계적 리팩터·보일러플레이트 위임 | 최종 판단 |
| agy | 대용량 컨텍스트 분석(전체 스캔·긴 문서), 아키텍처 관점 리뷰 | 파일 수정 |

## 토큰 효율 규칙
1. **tri-review는 ADR급 결정에만** — 되돌리기 어려운 것(UI 프레임워크, DB, AI provider, 동기화 프로토콜). 일상 코드 리뷰는 Claude 단독
2. **같은 대상 재리뷰 금지** — 실행 전 `docs/reviews/` 확인. 재리뷰가 필요하면 "이전 리뷰 이후 바뀐 부분"만 범위 지정
3. **범위를 파일 단위로 명시** — "repo 전체를 봐라"는 금지, 대상 파일 목록을 프롬프트에 적는다
4. **출력 형식 강제** — "칭찬 생략, 문제점만, 심각도 표기, N개 이내"를 프롬프트에 포함
5. **병렬 + 백그라운드** — codex/agy는 동시 실행하고 기다리는 동안 다른 작업
6. **결과는 즉시 본문 문서에 반영** — raw 출력을 다음 세션이 다시 읽게 만들지 않는다. `docs/reviews/`에는 판정 요약만
7. **양방향 질문** — 중요 리뷰는 "무엇이 부족한가"와 "무엇이 과한가"를 모두 묻는다 (한쪽만 물으면 편향된 답이 온다)
8. **CLAUDE.md는 얇게** — 매 세션 로드되므로 상세 규칙은 `docs/rules/`에 두고 CLAUDE.md는 포인터만

## 검증 의무
- 외부 엔진의 지적은 **원문·실제 파일 대조 후** 수용/기각 판정 (맹신 금지)
- 위험 플래그(`--dangerously-*`)는 사용자가 해당 대화에서 명시 허락한 경우만
